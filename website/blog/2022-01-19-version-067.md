## Destaques da 0.67

Como mencionado nos [planos H2 2021 da Meta](https://reactnative.dev/blog/2021/08/19/h2-2021), o React Native está tentando lançamentos mais frequentes para um tempo de resposta mais curto para novos recursos e correções (como a nova arquitetura) chegarem à comunidade. Naturalmente, muitos lançamentos focarão em correções e melhorias.

Aqui estão algumas mudanças notáveis chegando na 0.67.0:

- Remoções lean-core: [DatePickerAndroid](https://github.com/facebook/react-native/commit/7a770526c626e6659a12939f8c61057a688aa623#diff-e727e4bdf3657fd1d798edcd6b099d6e092f8573cba266154583a746bba0f346)
- Atualizar versão do Gradle para 7.2, Atualizar versão do Kotlin para 1.5.31 [Bump Kotlin and Gradle versions (#32319)](https://github.com/facebook/react-native/commit/9ae3367431428748f5486c782199beb4f9c6b477)
- Um destaque notável: 0.67 continua a depender do Hermes 0.9.0, inalterado da 0.66

Você pode encontrar o [changelog completo aqui](https://github.com/facebook/react-native/blob/main/CHANGELOG.md#v0670).

Você pode participar da conversa sobre o status deste lançamento [nesta discussão](https://github.com/reactwg/react-native-releases/discussions/10) - e, como sempre, para ajudá-lo a atualizar para esta versão, você pode usar o [upgrade helper](https://react-native-community.github.io/upgrade-helper/) ⚛️

## Melhorias no Processo de Lançamento

Como mencionado, o React Native tem reestruturado o pipeline de lançamento para permitir lançamentos mais frequentes, de modo que novos recursos e correções possam ser lançados mais rapidamente para a comunidade.

Nos últimos meses, abordamos alguns problemas que atrasam os lançamentos.

### Coordenação e Compartilhamento de Conhecimento

Investimos em nossa documentação de lançamentos para cobrir como executar um lançamento, FAQs, coordenação de issues de lançamento, etc. — tudo isso pode ser encontrado nesta seção da [wiki do react-native](https://github.com/facebook/react-native/wiki/Releases).
Com a documentação, os lançamentos não ficam mais bloqueados em nenhum indivíduo ou conhecimento tribal.

Além da documentação, também reformulamos a coordenação de lançamentos e movemos a discussão do status de pré-lançamento e patches para um grupo de discussão dedicado: [react-wg/react-native-releases](https://github.com/reactwg/react-native-releases/discussions).

### Clareza de Responsabilidade

Seguindo mais documentação, o trabalho de lançamento pode escalar de modo que nenhuma pessoa seja crítica para executar um lançamento.

Um lançamento do React Native é suscetível a um amplo espectro de potenciais pontos de falha e tem muitas dependências e acompanhamentos. Considerando que o uso do React Native varia pela comunidade, é essencial ter stakeholders envolvidos nos lançamentos. Definimos um conjunto de [papéis e responsabilidades no suporte a um lançamento](https://github.com/facebook/react-native/wiki/Release-Roles-and-Responsibilities).

### Sinal de Release Candidate

Outro problema com lançamentos é obter um bom sinal de que um lançamento não sofrerá de regressões de compilação. Isso pode ser resolvido com investimento crescente em teste de variantes de compilação, etc., mas o sinal de adoção continuará sendo útil por algum tempo.

No lançamento 0.67, pilotamos um programa "Release Tester" onde desenvolvedores React Native trabalhando em apps Open Source [se comprometem a testar release candidates](https://github.com/facebook/react-native/wiki/Release-Roles-and-Responsibilities#release-tester-responsibilities) em seus apps. Anteriormente, não havia expectativa formal de que a comunidade testaria release candidates para levantar quaisquer problemas potenciais. Este programa nos ajuda a obter sinal mais rápido para garantir um nível de estabilidade do lançamento.

Apps React Native open source são particularmente úteis devido à disponibilidade de código-fonte para ajudar a depurar quaisquer regressões. Com este programa em vigor, um testador de lançamento identificou uma regressão na 0.67 e conseguimos resolvê-la sem agitar a comunidade maior com um lançamento defeituoso.

## Interessado em ajudar o React Native a estabilizar novos lançamentos?

Uma ótima maneira de nos ajudar a detectar regressões é integrar a versão de pré-lançamento do React Native [`react-native@next`](https://www.npmjs.com/package/react-native) ou [`react-native@nightly`](https://www.npmjs.com/package/react-native) ao seu CI. Para quaisquer regressões, você pode [abrir uma issue de lançamento](https://github.com/facebook/react-native/issues/new?assignees=&labels=Needs%3A+Triage+%3Amag%3A%2CType%3A+Upgrade+Issue&template=upgrade-regression-form.yml) e notificar a discussão apropriada.

Se seu app ou empresa está interessado em se juntar ao programa "Release Tester", vá para a seção dedicada na parte inferior da [wiki de Papéis e Responsabilidades de Lançamento](https://github.com/facebook/react-native/wiki/Release-Roles-and-Responsibilities#release-tester-responsibilities) para saber mais.

Por último, qualquer ajuda em experimentar nossos release candidates ou ajudar a desbloquear issues de lançamento é muito apreciada!
