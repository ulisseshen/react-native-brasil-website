Após meses de trabalho árduo de centenas de contribuidores, a equipe principal do React Native tem orgulho de anunciar o lançamento da versão 0.60. Esta versão trata de migrações significativas para as plataformas Android e iOS, e muitos problemas também foram resolvidos. Este post do blog cobre os destaques do lançamento. Como sempre, consulte o changelog para informações mais detalhadas. Por fim, obrigado aos contribuidores por nos ajudarem a alcançar este marco!

## Foco em Acessibilidade

Houve muitas melhorias nas APIs de acessibilidade, como [announceForAccessibility](https://github.com/facebook/react-native/commit/cfe0032), além de melhorias em [roles](https://github.com/facebook/react-native/commit/1aeac1c), [suporte a ações](https://github.com/facebook/react-native/commit/14b4668), [flags](https://github.com/facebook/react-native/commit/0090ab3) e muito mais. Acessibilidade é uma ciência complexa, mas esperamos que essas melhorias facilitem um pouco ser A11Y. Não deixe de conferir [React Native Open Source Update June 2019](/blog/2019/06/12/react-native-open-source-update#meaningful-community-contributions) para mais detalhes sobre essas mudanças.

## Um Novo Começo

A tela inicial do React Native foi atualizada! Obrigado aos muitos contribuidores que ajudaram a criar a nova interface. Este novo "Hello World" dará as boas-vindas aos usuários ao ecossistema de uma maneira mais amigável e envolvente.

![A nova tela de inicialização ajuda os desenvolvedores a começar desde o início com recursos e um bom exemplo](/blog/assets/0.60-new-init-screen.png)

## Suporte ao AndroidX

[AndroidX](https://developer.android.com/jetpack/androidx) é um grande avanço no ecossistema Android, e os antigos artefatos da biblioteca de suporte estão sendo descontinuados. Para a versão 0.60, o React Native foi migrado para o AndroidX. Esta é uma mudança que quebra compatibilidade, e **seu código nativo e dependências também precisarão ser migrados**.

> Com esta mudança, os apps React Native precisarão começar a usar o AndroidX. Eles não podem ser usados lado a lado em um app, então todo o código do app e o código de dependências precisam estar usando um ou outro.
>
> [matt-oakes](https://github.com/matt-oakes) em [discussions-and-proposals](https://github.com/react-native-community/discussions-and-proposals/issues/129)

Embora seu próprio código nativo precise ser migrado por você, [@mikehardy](https://github.com/mikehardy), [@cawfree](https://github.com/cawfree) e [@m4tt72](https://github.com/m4tt72) construíram uma [ferramenta inteligente chamada "jetifier"](https://github.com/mikehardy/jetifier) para corrigir seus `node_modules`. Os mantenedores de bibliotecas precisarão atualizar, mas esta ferramenta fornece uma solução temporária enquanto dá a eles tempo para lançar uma versão AndroidX. Então, se você encontrar erros relacionados à migração do AndroidX, experimente isso.

## CocoaPods por Padrão

[CocoaPods agora fazem parte do projeto iOS do React Native](https://github.com/react-native-community/discussions-and-proposals/blob/master/proposals/0004-cocoapods-support-improvements.md). Se você ainda não estava fazendo isso, certifique-se de abrir o código da plataforma iOS usando o arquivo `xcworkspace` de agora em diante (dica: tente `xed ios` a partir do diretório raiz do projeto). Além disso, os `podspec`s para os pacotes internos mudaram para torná-los compatíveis com os projetos Xcode, o que ajudará na solução de problemas e depuração. Espere fazer [algumas mudanças diretas](https://github.com/facebook/react-native/commit/2321b3f) no seu `Podfile` como parte da atualização para 0.60 para trazer este suporte empolgante. Observe que estamos cientes de um problema de compatibilidade com `use_frameworks!`, e estamos acompanhando uma [issue](https://github.com/facebook/react-native/issues/25349) com soluções alternativas e um patch futuro.

## Remoções do Lean Core

**WebView** e **NetInfo** foram previamente extraídos para repositórios separados, e na versão 0.60 terminamos de migrá-los para fora do repositório do React Native. Além disso, em resposta ao feedback da comunidade sobre a nova política da App Store, **Geolocation** foi extraído. Se você ainda não o fez, complete sua migração adicionando dependências para [react-native-webview](https://github.com/react-native-community/react-native-webview), [@react-native-community/netinfo](https://github.com/react-native-community/react-native-netinfo) e [@react-native-community/geolocation](https://github.com/react-native-community/react-native-geolocation). Se você quiser uma solução automatizada, considere usar [rn-upgrade-deprecated-modules](https://github.com/lucasbento/rn-update-deprecated-modules). Os mantenedores fizeram mais de 100 commits nesses repositórios desde a extração e estamos empolgados em ver o apoio da comunidade!

## Native Modules agora são Autovinculados

A equipe que trabalha no [React Native CLI](https://github.com/react-native-community/cli) introduziu grandes melhorias na vinculação de native modules chamadas [autolinking](https://github.com/react-native-community/cli/blob/master/docs/autolinking.md)! A maioria dos cenários não exigirá mais o uso de `react-native link`. Ao mesmo tempo, a equipe renovou o processo de vinculação em geral. Certifique-se de usar `react-native unlink` em quaisquer dependências pré-existentes, conforme mencionado nos documentos acima.

## Upgrade Helper

[@lucasbento](https://github.com/lucasbento), [@pvinis](https://github.com/pvinis), [@kelset](https://github.com/kelset) e [@watadarkstar](https://github.com/watadarkstar) construíram uma ótima ferramenta chamada [Upgrade Helper](https://react-native-community.github.io/upgrade-helper/) para tornar o processo de atualização mais simples. Ela ajuda usuários do React Native com brownfield apps ou customizações complexas a ver o que mudou entre versões. Dê uma olhada na [documentação de atualização atualizada](/docs/upgrading) e experimente hoje mesmo para seu caminho de atualização!

![Upgrade Helper mostra de forma clara e fácil as mudanças necessárias para migrar para uma versão diferente do React Native](/blog/assets/0.60-upgrade-helper.png)

## Uma Nota para Mantenedores de Bibliotecas

As mudanças para AndroidX quase certamente exigirão atualizações em sua biblioteca, então certifique-se de incluir suporte em breve. Se você não puder atualizar ainda, considere verificar sua biblioteca com o jetifier para confirmar que os usuários podem corrigir sua biblioteca em tempo de compilação.

Revise a documentação de [autolinking](https://github.com/react-native-community/cli/blob/master/docs/autolinking.md) para atualizar suas configurações e readme. Dependendo de como sua biblioteca foi previamente integrada, você também pode precisar fazer algumas mudanças adicionais. Confira o guia de [dependencies](https://github.com/react-native-community/cli/blob/master/docs/dependencies.md) da CLI para informações sobre como definir a interface de sua dependência.
