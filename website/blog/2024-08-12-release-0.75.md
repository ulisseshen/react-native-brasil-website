## Destaques

### Yoga 3.1 e Melhorias de Layout

Desde que lançamos a Yoga [versão 3.0](https://reactnative.dev/blog/2024/04/22/release-0.74#yoga-30) no React Native 0.74, continuamos adicionando muitas melhorias e novas capacidades de layout para suas aplicações. O React Native 0.75 vem com Yoga 3.1 e você pode aprender mais sobre as novidades no [post oficial de lançamento](https://www.yogalayout.dev/blog/announcing-yoga-3.1) da Yoga.

Uma funcionalidade notável e muito solicitada é o suporte para valores `%` em vários lugares, como `gaps` e `translation`

:::info
Essas funcionalidades estão disponíveis apenas para a New Architecture. Se você deseja usá-las, por favor considere migrar para ela.
:::

#### Valores em Porcentagem em Gaps

Com 0.75, as props `gap`, `columnGap` e `rowGap` descritas [aqui](https://reactnative.dev/docs/flexbox#row-gap-column-gap-and-gap) agora suportam uma string com um valor `%`.

Por exemplo:

```tsx
function App(): React.JSX.Element {
  return (
    <SafeAreaView
      style={{
        marginTop: 20,
        alignItems: 'center',
        flex: 1,
        rowGap: '20%',
      }}>
      <View
        style={{flex: 1, flexDirection: 'row', columnGap: '10%'}}>
        <View
          style={{
            backgroundColor: 'purple',
            width: 100,
            height: 100,
          }}
        />
        <View
          style={{
            backgroundColor: 'blue',
            width: 100,
            height: 100,
          }}
        />
        <View
          style={{
            backgroundColor: 'green',
            width: 100,
            height: 100,
          }}
        />
      </View>
      <View
        style={{flex: 1, flexDirection: 'row', columnGap: '10%'}}>
        <View
          style={{
            backgroundColor: 'lime',
            width: 100,
            height: 100,
          }}
        />
        <View
          style={{
            backgroundColor: 'yellow',
            width: 100,
            height: 100,
          }}
        />
        <View
          style={{
            backgroundColor: 'orange',
            width: 100,
            height: 100,
          }}
        />
      </View>
      <View
        style={{flex: 1, flexDirection: 'row', columnGap: '10%'}}>
        <View
          style={{
            backgroundColor: 'red',
            width: 100,
            height: 100,
          }}
        />
        <View
          style={{
            backgroundColor: 'violet',
            width: 100,
            height: 100,
          }}
        />
        <View
          style={{
            backgroundColor: 'magenta',
            width: 100,
            height: 100,
          }}
        />
      </View>
    </SafeAreaView>
  );
}
```

#### Valores em porcentagem em Translation

A [prop `transform`](https://reactnative.dev/docs/transforms) agora também pode aceitar `%` como valores para as transformações `translate`.

Por exemplo, o seguinte componente moveria a coordenada X do quadrado vermelho em 100% de sua largura e a coordenada Y em 100% de sua altura:

```tsx
function Translated() {
  return (
    <SafeAreaView
      style={{
        marginTop: 20,
        flex: 1,
        rowGap: '20%',
      }}>
      <View
        style={{
          backgroundColor: 'red',
          width: 100,
          height: 100,
          transform: [{translateY: '100%'}, {translateX: '100%'}],
        }}
      />
    </SafeAreaView>
  );
}
```

### Estabilização da New Architecture

Desde nosso anúncio da [New Architecture em Beta](https://github.com/reactwg/react-native-new-architecture/discussions/189) na React Conf, entregamos várias correções de bugs e melhorias para sua estabilidade.

Nosso objetivo é que a New Architecture seja considerada estável no futuro próximo. Portanto, nos últimos meses focamos em preencher as lacunas entre a Old e a New Architecture. Alguns exemplos de bugs e funcionalidades faltantes que corrigimos são:

- Corrigir `adjustsFontSizeToFit` no Android ([#44075](https://github.com/facebook/react-native/pull/44075))
- Corrigir `textAlign` não funcionando com views inline no Android ([#44146](https://github.com/facebook/react-native/pull/44146))
- Corrigir baseline de texto sendo movida para cima no iOS ([#44932](https://github.com/facebook/react-native/pull/44932))

Junto com o pessoal da Expo, também trabalhamos em adicionar informações sobre suporte à New Architecture no [React Native Directory](https://reactnative.directory/), então ficará imediatamente claro para você se uma biblioteca já suporta a New Architecture ou não:

![React Native Directory](../static/blog/assets/0.75-rn-directory.png)

Também convidamos você a participar da [pesquisa sobre New Architecture](https://t.co/ucAA58hnFu). Esta pesquisa é fundamental para coletarmos feedback precioso sobre os próximos passos para o lançamento da New Architecture.

Também queremos compartilhar um post que publicamos no Working Group da New Architecture sobre [Suporte ao UIManager na New Architecture](https://github.com/reactwg/react-native-new-architecture/discussions/201). Este post oferece uma visão geral da API do `UIManager` no Android e como ela pode ajudar na migração de apps e bibliotecas mais avançados.

Este lançamento também contém uma nova API que agora é a maneira recomendada de acessar o `jsi::Runtime`.

#### Acessando `jsi::Runtime` em TurboModules

No passado, nunca houve uma maneira recomendada dos módulos nativos acessarem o `jsi::Runtime`, e os consumidores contornavam o framework para fazê-lo de maneiras arriscadas. Em 0.74, introduzimos APIs experimentais fornecendo acesso seguro ao `jsi::Runtime`, e estamos felizes em anunciar sua estabilidade para 0.75.

<details>
<summary>Exemplos de como acessar o jsi::Runtime</summary>

No iOS, você pode fazer seu Turbo Native Module estar em conformidade com o protocolo `RCTTurboModuleWithJSIBindings`. Você pode agora implementar `installJSIBindingsWithRuntime`, que lhe dará acesso thread-safe ao runtime.

```objc
@interface RCTSampleTurbo Module () <RCTTurboModuleWithJSIBindings>
@end

#pragma mark - RCTTurboModuleWithJSIBindings
- (void)installJSIBindingsWithRuntime:(jsi::Runtime &)runtime {
  runtime.global().setProperty(
    runtime,
    "myGlobalFunction",
    jsi:: Function::createFromHost Function(...));
}
```

No Android, você pode fazer seu Turbo Native Module estar em conformidade com a interface `TurboModuleWithBindings`. Você pode agora implementar o método JNI `getBindingsInstaller`, que lhe dará acesso thread-safe ao runtime em C++.

```java
public class SampleTurboModule extends NativeSampleTurboModuleSpec implements TurboModuleWithJSIBindings

@Override
public native BindingsInstallerHolder getBindingsInstaller();
```

```cpp
// C++
jni::local_ref<BindingsInstallerHolder::javaobject> SampleTurboModuleJSIBindings::getBindingsInstaller(jni::alias_ref<jni::object> jobj) {
  return BindingsInstallerHolder::newObjectCxxArgs(
    [](jsi::Runtime& runtime) {
      runtime.global().setProperty(
        runtime,
        "myGlobalFunction",
        jsi::Function::createFromHostFunction(...));
    }
  );
}
```

Se você está na UI thread e precisa acessar o runtime, introduzimos uma nova API: `CallInvoker`. Ela consiste em um único método, `invokeAsync`, que irá pular para a JS thread para executar com segurança algum trabalho usando o JS runtime. Essas APIs são compatíveis com versões futuras.

No iOS, fornecemos o protocolo `RCTCallInvokerModule`. Após estar em conformidade com este protocolo, nossa infraestrutura decorará o módulo com acesso ao `CallInvoker`.

```objc
@interface RCTSampleTurboModule() <RCTCallInvokerModule>

[self.callInvoker callInvoker].invokeAsync([&](jsi::Runtime& runtime) {
  // do stuff on JS thread
}
```

No Android, o `CallInvoker` é acessível através do `ReactContext` em um wrapper JNI chamado `CallInvokerHolder`, onde você pode então chamar `invokeAsync` depois de cruzar a fronteira JNI.

```java
// Java
public abstract CallInvokerHolder getJSCallInvokerHolder();
```

```cpp
// C++
jsCallInvokerHolder->cthis()->getCallInvoker()->invokeAsync([&](jsi::Runtime& rt) {
  // do stuff on JS thread
});
```

</details>

### Usando Frameworks

Como compartilhamos na React Conf no início deste ano, a maneira recomendada de construir um app React Native agora é através de um framework, como Expo.

Você pode ler mais sobre esta orientação em nosso post anterior no blog: "[Use um framework para construir apps React Native](https://reactnative.dev/blog/2024/06/25/use-a-framework-to-build-react-native-apps)".

Queremos preparar novos usuários do React Native para o sucesso. Acreditamos que usar um framework torna você o mais produtivo possível, e oferece a melhor experiência de desenvolvedor ao construir novos apps.

Para refletir essas recomendações, esta versão inclui as seguintes mudanças:

- Movemos a pasta `/template` do pacote `react-native` para um repositório separado: [`react-native-community/template`](https://github.com/react-native-community/template).
- Estamos descontinuando o comando `react-native init` a partir de 31 de dezembro de 2024.

Se você já está usando um framework como Expo, essas mudanças não irão impactá-lo de forma alguma. Você poderá usar React Native 0.75 junto com Expo SDK 51 (você pode encontrar instruções sobre como fazer isso em [este post dedicado da Expo](https://expo.dev/changelog/2024/08-14-react-native-0.75)).

Se você não está usando um framework ou está construindo seu próprio framework, vamos ver como essas mudanças irão impactá-lo.

#### Movendo o template para react-native-community/template

Historicamente, `react-native` costumava entregar uma pasta `/template` dentro do pacote NPM que era usada pela Community CLI para criar novos projetos. Isso tornava a atualização do template bastante lenta, pois precisávamos de um novo lançamento do React Native para cada mudança no template.

Com nossa última recomendação de usar um framework, sentimos que entregar um template opinativo dentro de nosso pacote NPM core não estava alinhado com [nossa visão](https://github.com/react-native-community/discussions-and-proposals/blob/main/proposals/0759-react-native-frameworks.md).

Portanto, decidimos mover o template para o pacote [`@react-native-community/template`](https://www.npmjs.com/package/@react-native-community/template).

Isso tornará mais fácil para a comunidade manter e evoluir o template, sem ter que depender de um lançamento do React Native para cada mudança. Além disso, isso aproxima o template da Community CLI e tornará mais fácil para todos inspecionar e depender do template como um pacote separado.

Esta mudança deve ser completamente transparente para usuários que criam novos projetos usando a Community CLI. A partir de agora, novos issues relacionados ao template devem ser reportados no [rastreador de issues do template](https://github.com/react-native-community/template/issues?q=sort%3Aupdated-desc+is%3Aissue+is%3Aopen). Todas as várias ferramentas que dependem do template, como o upgrade-helper, também foram atualizadas de acordo e continuarão funcionando como de costume.

#### Descontinuando react-native init

Similarmente ao template, o comando `react-native init` também foi adaptado para se alinhar com a nova recomendação.

Historicamente, `react-native init` era o comando padrão para criar novos projetos React Native. No entanto, em 2024, sentimos que este comando não fornece a mesma experiência de onboarding que um framework ofereceria. É por isso que não o estamos mais recomendando, em vez disso você deve usar um framework como Expo.

Você ainda pode usar `react-native init` para criar novos projetos usando a Community CLI e template hoje, mas você verá o seguinte aviso:

![Init Deprecation](../static/blog/assets/0.75-deprecation.png)

A partir de 31 de dezembro de 2024, o comando `init` não criará mais um projeto. Você terá que:

- Usar um framework como Expo, com seu próprio comando dedicado para criar um novo projeto, como `npx create-expo-app`
- Invocar a Community CLI diretamente com `npx @react-native-community/cli init`

Por favor, note que `react-native config` e todos os outros comandos além de `init` continuarão funcionando como de costume.

:::info
Para oferecer uma experiência de migração mais suave, o pacote `react-native@0.75.0` ainda está dependendo de `@react-native-community/cli`, mas estamos planejando remover esta dependência no futuro próximo.
:::

#### Melhorias de desempenho do auto-linking

Durante este trabalho de atualização do comando `init`, também passamos tempo reescrevendo a lógica de auto-linking para ser mais performática. Isso resulta em velocidade de build mais rápida tanto para Android quanto para iOS.

Com React Native 0.75, se você estiver usando Expo, o passo de auto-linking pode agora rodar ~6.5x mais rápido no Android e ~1.5x mais rápido no iOS. Você pode ler mais sobre essas melhorias [aqui](https://github.com/react-native-community/discussions-and-proposals/discussions/814).

### Breaking changes

Embora esta próxima seção pareça extensa, esperamos que as breaking changes aqui irão impactar principalmente um pequeno grupo de usuários que estão usando React Native de maneiras mais avançadas.

Queremos apresentá-las aqui por uma questão de completude e para referência.

#### Touchables em TypeScript não podem mais ser usados como tipos em expressões Genéricas

Componentes `TouchablesOpacity` e `TouchableHighlights` foram convertidos para componentes funcionais. Isso significa que eles não podem mais ser usados como `value & type`. Então, por exemplo, o seguinte não é mais válido:

```tsx
import {TouchableHighlight} from 'react-native';
const ref = useRef<TouchableHighlight>();
//                ^^^ TS2749: TouchableHighlight refers to a value, but is being used as a type here.
//                            Did you mean typeof TouchableHighlight?
```

Em vez disso, você deve usar o tipo React embutido `React.ElementRef` ou, alternativamente, o tipo `View`:

```tsx
import {TouchableHighlight} from 'react-native';
const ref1 =
  useRef<React.ElementRef<typeof TouchableHighlight>>();
// or
const ref2 = useRef<View>();
```

#### Última versão com suporte para minSdk 23 e minIOSVersion 13.4

Estas não são breaking changes em 0.75 _per se_, mas queremos compartilhar que React Native 0.75 será a última versão do React Native a suportar minSdk 23 (Android 6.0) e minIOSVersion 13.4.

A partir do React Native 0.76, a versão minSdk será 24 (Android 7.0) e a minIOSVersion será 15.1.

Você pode ler mais sobre isso em nosso anúncio oficial [para Android](https://github.com/react-native-community/discussions-and-proposals/discussions/802) e [para iOS](https://github.com/react-native-community/discussions-and-proposals/discussions/812).

#### Android: JSIModule foi deletado

O `com.facebook.react.bridge.JSIModule` ([source](https://github.com/facebook/react-native/blob/0.73-stable/packages/react-native/ReactAndroid/src/main/java/com/facebook/react/bridge/JSIModule.java)) era uma API que introduzimos temporariamente para permitir que um Native Module acessasse JSI diretamente no Android.
Os acessores para esta API [foram depreciados em 0.74](https://reactnative.dev/blog/2024/04/22/release-0.74#other-breaking-changes), e verificamos que não havia uso significativo desta API em Open Source, então estamos removendo-a em 0.75.
Você pode usar [Turbo Native Modules](https://github.com/reactwg/react-native-new-architecture/blob/main/docs/turbo-modules.md) como alternativa.

#### Android: PopUp Menu movido para pacote separado

Em 0.74, [movemos o `PopUpMenu` do Android para um pacote separado](https://reactnative.dev/blog/2024/04/22/release-0.74#other-breaking-changes) sob o escopo `@react-native`.
Em 0.75, estamos removendo os métodos restantes que ainda estavam no core:

- `UIManagerModule.showPopupMenu()`
- `UIManagerModule.dismissPopupMenu()`

Como alternativa, por favor use o componente `<PopupMenuAndroid />`, que vive no pacote [`@react-native/popup-menu-android`](https://www.npmjs.com/search?q=%40react-native%2Fpopup-menu-android).

#### iOS: Finalizado trabalho de depreciação de PushNotificationIOS

Em 0.74, [depreciamos](https://reactnative.dev/blog/2024/04/22/release-0.74#api-changes-to-pushnotificationios-deprecated) algumas APIs do módulo `PushNotificationIOS`.

Em 0.75, deletamos essas APIs para migrar de representações legadas de metadados de notificações.

As APIs que foram deletadas são:

```objc
  + (void)didReceiveLocalNotification:(UILocalNotification *)notification;
  + (void)didReceiveRemoteNotification:(NSDictionary *)notification;
```

Em vez disso, use `didReceiveNotification:(UNNotification *)notification`.

#### Community CLI: Remoção dos comandos ram-bundle e profile-hermes

Gostaríamos de anunciar duas remoções significativas vindas da Community CLI: os comandos `ram-bundle` e `profile-hermes`.

O comando `ram-bundle` foi introduzido no React Native 0.59 para permitir que você executasse seus bundles carregando-os diretamente na memória. Esta funcionalidade agora é substituída pela Hermes, nosso engine JS padrão. Você não deve usar o comando `ram-bundle`.

O comando `profile-hermes` era uma ferramenta para ajudá-lo a perfilar o desempenho de CPU do seu código JavaScript. Isso usava o antigo formato `.cpuprofile`, que não é mais suportado em versões recentes do Chrome. Incluir esta capacidade como um comando standalone também é algo do qual estamos nos afastando à medida que trabalhamos em elevar a barra de qualidade de nossas ferramentas de debugging. Profiling de CPU agora pode ser acessado a partir do painel "Profiler" no [Experimental New Debugger](https://reactnative.dev/docs/0.74/debugging?js-debugger=new-debugger#opening-the-debugger) (Nota: isso não é acessível se conectando à Hermes a partir do Chrome).

## Outras Breaking changes

### Geral

- **Codegen**
  - Mudou ligeiramente o nome das classes e structs geradas de TurboModules C++ puros. Removemos o token `Cxx` de seus nomes
  - Enums float não são mais suportados devido a possíveis erros de precisão
  - Lança um erro ao passar `null` em JS para um argumento não anulável em Native
- **Linting**
  - Config do ESLint não executa mais prettier ao fazer linting
- **C++**
  - `ScrollViewShadowNode` agora requer um novo parâmetro `bool includeTransform` no construtor
  - Removido `executeAsynchronously` e `executeSynchronously_CAN_DEADLOCK` de RuntimeExecutor
  - Renomeado `JsErrorHandlingFunc` para `OnJsError` em `JsErrorHandler.h`
  - Renomeado `handleJsError` para `OnJsError` em `handleFatalError.h`
  - Removido `import` não utilizado de `ReactPrimitives.h`
  - Métodos get de `LongLivedObjectCollection` e `LongLivedObject` agora aceitam um parâmetro Runtime
  - Renomeado o arquivo `utils/jsi.h` para `jsi-utils.h`
- **TextInput**
  - Removido callback depreciado `onTextInput`
- **Pressability**
  - Removido método `onLongPressShouldCancelPress_DEPRECATED`, `onResponderTerminationRequest_DEPRECATED`, e `onStartShouldSetResponder_DEPRECATED`

#### Android

- **ReactViewBackgroundDrawable**
  - Depreciado em favor de `CSSBackgroundDrawable`. Isso também remove algumas APIs de `ReactViewBackgroundDrawable` e de `ColorUtil`
- **ReactContext**
  - `ReactContext` e `ReactApplicationContext` agora são abstratos. Use `BridgeReactContext` e `BridgelessReactContext` no lugar
  - Delete `ReactContext.initializeWithInstance()`. Por favor use `BridgeReactInstance` no lugar
  - Remover `BridgelessReactContext.getJavaScriptContextHolder()`. Por favor use `BridgelessCatalystInstance` no lugar
  - Remover `ReactContext.getRuntimeExecutor()`. Por favor use `BridgelessCatalystInstance`
- **Layout**
  - Suporte para valores de flex gap em porcentagem. Isso muda os parâmetros de alguns métodos como `setGap`, `setRowGap` e `setColumnGap` de float para dynamic
  - Requer `supportsRTL` no Android Manifest
- **Runtime**
  - Removido `ReactJsExceptionHandler` de ReactHostImpl
  - Torna o app responsável por retornar os core turbomodules quando não estiver usando o template padrão
- **DevSupport**
  - Mudado o `DevSupportManagerFactory.create()` para aceitar um novo parâmetro `PausedInDebuggerOverlayManager`
- **Measurement**
  - Deletado `UIManagerModule.measureLayoutRelativeToParent()`

#### iOS

- **Runtime**
  - Remover `[RCTHost getSurfacePresenter]` e `[RCTHost getModuleRegistry]`
  - Remover classe `EventPriority` e sempre usar o padrão `EventPriority::AsynchronousBatched`. Se o build falhar, por favor remova qualquer uso de `EventPriority`
- **Image**
  - Remover `RCTImageLoadingPerfInstrumentationEnabled` não utilizado
- **Error Handling**
  - Remover acesso ao `RCTRedBox` através de `RCTBridge`
- **CocoaPods**
  - Renomeado `BUILD_FROM_SOURCE` para `RCT_BUILD_HERMES_FROM_SOURCE`
  - Renomeado `React-Codegen` para `ReactCodegen` para melhor compatibilidade com `use_frameworks` e Swift
- **TextInput**
  - Remover callback depreciado `onTextInput`
