## Destaques

### Melhorias na Depuração

As equipes do React Native e Hermes estão comprometidas em melhorar a experiência de depuração no React Native. Na versão 0.73, temos o prazer de compartilhar alguns progressos iniciais deste investimento contínuo.

#### Histórico de Console Log no Hermes

`console.log()` é uma maneira sempre popular para desenvolvedores depurarem rapidamente seu código JavaScript. Em versões anteriores, logs de console em aplicativos React Native não eram gravados até que um depurador fosse conectado. Isso dificultava a observação de logs que ocorrem no início durante o carregamento do aplicativo.

No React Native 0.73 resolvemos este problema. O Hermes agora captura todas as chamadas `console.log()` em segundo plano, que serão enviadas para a aba Console quando um depurador for conectado pela primeira vez — correspondendo à experiência de depuração em navegadores web. Este novo comportamento funciona no Flipper, Chrome DevTools conectado ao Hermes e no novo Debugger experimental.

#### Documentação de Depuração Atualizada

Atualizamos a seção [Debugging](/docs/debugging) da nossa documentação, que agora inclui informações atualizadas sobre como conectar todos os depuradores suportados, mais informações sobre o React DevTools e visuais renovados.

![Visão geral da documentação de Debugging na versão 0.73](/blog/assets/0.73-debugging-docs.jpg)

#### Novo Debugger Experimental

A equipe do React Native está trabalhando em uma nova experiência de depurador JavaScript, destinada a substituir o Flipper, com uma Technical Preview disponível a partir do React Native 0.73. O novo depurador abre imediatamente e apresenta uma interface simplificada do Chrome DevTools personalizada para depuração do React Native com Hermes.

:::note
O novo depurador é **experimental** e tem alguns [problemas conhecidos](https://github.com/react-native-community/discussions-and-proposals/discussions/733) nos quais estamos trabalhando ativamente para resolver em uma versão futura do React Native. Se você está experimentando, por favor use a [mesma thread de discussão](https://github.com/react-native-community/discussions-and-proposals/discussions/733) para reportar feedback.
:::

![O novo frontend do depurador aberto no painel "Welcome"](/docs/assets/debugging-debugger-welcome.jpg)

Saiba mais sobre como habilitar esta experiência [na documentação](/docs/next/debugging?js-debugger=new-debugger#opening-the-debugger).

### Suporte Estável a Symlink no Metro

O suporte para resolução de symlinks no Metro agora está **habilitado por padrão**. O suporte a symlink permite que o React Native funcione com configurações de monorepo quando os diretórios contidos são configurados com [`watchFolders`](https://metrobundler.dev/docs/configuration/#watchfolders).

Os symlinks são profundamente representados nos componentes internos do Metro, o que significa que funcionam com recursos como [Fast Refresh](/docs/fast-refresh), e incorrem em pouca sobrecarga de desempenho durante o bundling. Os symlinks são suportados em todas as plataformas desktop, com e sem Watchman.

:::info

#### Soluções alternativas para monorepo

Estamos cientes de que ainda existem casos extremos ao usar o React Native em um layout de monorepo. Temos trabalho planejado para resolver alguns deles, que não foram incluídos na versão 0.73 — mas que pretendemos lançar o mais rápido possível.

**Para projetos de template React Native (`npx react-native init`)**, você precisará configurar quaisquer [`watchFolders`](https://metrobundler.dev/docs/configuration/#watchfolders) fora da raiz do projeto para que o Metro os descubra ([mais informações](https://metrobundler.dev/docs/configuration/#unstable_enablesymlinks-experimental)). Você também pode precisar atualizar caminhos de arquivo se sua dependência `react-native` estiver instalada em uma pasta em um nível diferente.

**Para aplicativos Expo**, o suporte para workspaces Yarn (Classic) é configurado por padrão. Veja também o guia [Work with monorepos](https://docs.expo.dev/guides/monorepos/) na documentação do Expo.
:::

### Template Kotlin no Android

Estamos empolgados em anunciar que, a partir da versão 0.73, Kotlin agora é a **linguagem recomendada** para aplicativos Android construídos com React Native. Isso segue a direção que o ecossistema Android vem tomando há vários anos e permite que você escreva seu aplicativo usando uma linguagem moderna.

Atualizamos o template do React Native no Android para usar Kotlin em vez de Java. Os novos arquivos `MainActivity.kt` e `MainApplication.kt` são 36% menores em tamanho.

O [Upgrade Helper](https://react-native-community.github.io/upgrade-helper/) também foi atualizado para facilitar a migração de seus arquivos `.java` para arquivos `.kt`.
Se você modificou anteriormente os arquivos Java em seu projeto e precisa de suporte para migrá-los para Kotlin, você pode usar o utilitário `Code > Convert Java file to Kotlin File` do Android Studio (também acessível com o atalho <kbd>Cmd ⌘</kbd> + <kbd>Shift ⇧</kbd> + <kbd>Option ⌥</kbd> + <kbd>K</kbd>).

### Suporte ao Android 14

Atualizamos o React Native para oferecer suporte total ao Android 14. A partir da versão 0.73, desenvolvedores React Native agora podem ter como alvo a versão mais recente do SDK Android, [API Level 34](https://developer.android.com/guide/topics/manifest/uses-sdk-element?hl=en#ApiLevels) (_Upside Down Cake_).

#### Atualização do Java 17 e Android Gradle Plugin

Para suportar o Android 14, atualizamos a versão do Android Gradle Plugin (AGP) usado para construir aplicativos Android de `7.4.x` para `8.1.x`.

Este grande aumento de versão do AGP vem com uma série de mudanças incompatíveis que estão disponíveis nas notas de lançamento do Google ([8.0.0](https://developer.android.com/build/releases/past-releases/agp-8-0-0-release-notes) e [8.1.0](https://developer.android.com/build/releases/past-releases/agp-8-1-0-release-notes)).

Mais importante, **Java 17** agora é um requisito para construir aplicativos Android. Você pode atualizar sua versão do Java para 17 executando:

```sh
brew install --cask zulu@17
```

e atualizando seu `JAVA_HOME` conforme documentado no [guia Getting Started](https://reactnative.dev/docs/environment-setup).

Se você é desenvolvedor de biblioteca, suas bibliotecas devem funcionar com React Native 0.73 sem alterações. No início deste ano, [publicamos uma nota](https://github.com/react-native-community/discussions-and-proposals/issues/671) com um esclarecimento sobre o que o aumento do AGP significa para você como autor de biblioteca.

#### Conceder acesso parcial a fotos e vídeos

[Selected Photos Access](https://developer.android.com/about/versions/14/changes/partial-photo-video-access) permite que usuários do Android 14 concedam aos aplicativos acesso a itens específicos em sua biblioteca de mídia, em vez de acesso a toda a mídia. Na versão 0.73, aplicativos React Native agora suportam essa capacidade, usando a permissão `READ_MEDIA_VISUAL_USER_SELECTED` na API [`PermissionsAndroid`](/docs/permissionsandroid).

![Selected Photos Access no Android 14](/blog/assets/0.73-android-media-picker.jpg)

#### Aumento do Min SDK

React Native 0.73 será a versão final que suporta Android 5.0 (API Level 21). A próxima versão do React Native terá uma versão mínima de SDK de 23 (Android 6.0). Leia mais sobre as próximas mudanças no aumento do min SDK [aqui](https://github.com/react-native-community/discussions-and-proposals/discussions/740).

### Atualizações da Nova Arquitetura

Continuamos a implantação da Nova Arquitetura do React Native, para disponibilizá-la a todos na comunidade Open Source.

Desde o React Native 0.68, tanto o New Renderer (Fabric) quanto o New Native Module System (TurboModules) estavam disponíveis para os usuários experimentarem e avaliarem. Queremos agradecer à comunidade pelo feedback que recebemos até agora.

Hoje estamos lançando outra peça da Nova Arquitetura: [**Bridgeless Mode**](https://github.com/reactwg/react-native-new-architecture/discussions/154). Até agora, quando você habilita a Nova Arquitetura em seu aplicativo, o Bridge ainda estaria disponível para suportar compatibilidade retroativa com componentes e módulos mais antigos. No entanto, nossa visão é encerrar completamente o bridge. A partir do React Native 0.73, você pode habilitar o Bridgeless Mode, que desabilitará completamente a criação do bridge.

Junto com o Bridgeless Mode, estamos lançando uma Native Module Interop Layer, que permitirá que você reutilize seus módulos antigos quando estiver no Bridgeless Mode. A [Renderer Interop Layer](https://github.com/reactwg/react-native-new-architecture/discussions/135) introduzida no React Native 0.72 também foi adaptada para funcionar com o Bridgeless Mode.

Assim como o resto da Nova Arquitetura, o Bridgeless Mode é inicialmente experimental. Convidamos usuários interessados a habilitá-lo e reportar quaisquer problemas e incompatibilidades que você encontrar no [grupo de trabalho da Nova Arquitetura](https://github.com/reactwg/react-native-new-architecture/discussions).

### Recursos de Depuração Descontinuados

#### Integração Flipper ↔ React Native

Estamos gradualmente nos afastando do Flipper como a ferramenta padrão para depuração de aplicativos React Native. Na versão 0.73, isso começa com a descontinuação da integração nativa do Flipper incluída com o React Native (código de inicialização que conecta os plugins principais do Flipper). Removeremos esta integração e dependência na próxima versão — o que significa que partes do Flipper, como o plugin Network, deixarão de funcionar.

**Inalterado**: O Flipper como um produto autônomo para depuração de aplicativos nativos continuará a existir. Mesmo após sua remoção de novos projetos React Native no futuro, os desenvolvedores poderão [adicionar manualmente o Flipper ao seu aplicativo](https://fbflipper.com/docs/getting-started/android-native/) se desejarem.

Para mais informações sobre por que estamos nos afastando do Flipper, [veja o RFC](https://github.com/react-native-community/discussions-and-proposals/blob/main/proposals/0641-decoupling-flipper-from-react-native-core.md).

#### Remote JavaScript Debugging

Remote JavaScript Debugging é um modo de depuração legado que conecta um navegador web externo (Chrome) ao seu aplicativo e executa seu código JavaScript dentro de uma página web, ou seja, `http://localhost:8081/debugger-ui`. Este modelo pode levar a comportamento inconsistente do aplicativo durante a depuração e é incompatível com módulos nativos sob a Nova Arquitetura.

Na versão 0.73, [Remote JavaScript Debugging está descontinuado](https://github.com/react-native-community/discussions-and-proposals/discussions/734) e foi removido do Dev Menu. Habilitar o depurador remoto agora deve ser feito manualmente via API `NativeDevSettings`. Isso é abordado na [documentação de Outros Métodos de Depuração](/docs/next/other-debugging-methods#remote-js-debugging).

:::info
Remote JavaScript Debugging era anteriormente a experiência de depuração padrão para aplicativos usando JavaScriptCore (JSC). Recomendamos [Safari Developer Tools (depuração direta JSC)](/docs/next/other-debugging-methods#safari-developer-tools-direct-jsc-debugging) em vez disso, para aplicativos iOS.

Recomendamos usar [Hermes](/docs/hermes) para uma experiência de depuração consistente em todas as plataformas.
:::

## Mudanças Incompatíveis

### Renomeação de Pacotes Babel

Realocamos dois pacotes relacionados ao Babel para fora do Metro e para o repositório e esquema de versionamento do React Native, permitindo-nos simplificar a manutenção e atualizações. As novas versões desses pacotes suportam recursos da Nova Arquitetura na versão 0.73, o que significa que essas dependências devem ser atualizadas.

Por favor, siga o [Upgrade Helper](https://react-native-community.github.io/upgrade-helper/) ao atualizar, para garantir que você atualizou essas dependências. Alguns pacotes foram renomeados:

| Nome do Pacote Antigo                  | Nome do Pacote Novo                     |
| -------------------------------------- | --------------------------------------- |
| `metro-react-native-babel-preset`      | `@react-native/babel-preset`            |
| `metro-react-native-babel-transformer` | `@react-native/metro-babel-transformer` |

:::info
`@react-native/babel-preset` agora inclui `@react-native/babel-plugin-codegen`, isso não precisa mais ser especificado separadamente em seu arquivo de configuração Babel.
:::

### Outras Mudanças Incompatíveis

Estas são algumas das principais mudanças incompatíveis na versão 0.73. Por favor, consulte o [changelog completo](https://github.com/facebook/react-native/blob/main/CHANGELOG.md) para a lista completa de mudanças incompatíveis.

- Aumento do requisito mínimo do Node.js para 18.x ([#37709](https://github.com/facebook/react-native/pull/37709)) (veja também [Node.js 16 EOL](https://nodejs.org/en/blog/announcements/nodejs16-eol)).
- O template agora usa TypeScript 5.0 ([#36862](https://github.com/facebook/react-native/pull/36862)).
  - Os tipos do React Native continuam funcionando no TypeScript 4.8.
- **Android**: Java 17 agora é um requisito para construir aplicativos Android ([veja acima](#java-17-and-android-gradle-plugin-upgrade)).
- **Android**: Grande atualização do Fresco para 3.0 ([#38275](https://github.com/facebook/react-native/pull/38275)).
- **iOS**: Aumento da versão mínima do iOS para 13.4 ([#36795](https://github.com/facebook/react-native/pull/36795)).
- **iOS**: O Metro não será mais iniciado automaticamente ao executar builds via Xcode ([#38242](https://github.com/facebook/react-native/pull/38242)).

Para autores de biblioteca:

- **Android**: Atualização para AGP 8.1.1 ([discussão](https://github.com/react-native-community/discussions-and-proposals/issues/671))

### Mudanças no React Native CLI

#### Mudanças incompatíveis destacadas

- Alteração do prefixo de tarefa padrão no comando `build-android`. A partir de agora, quando você executar `build-android`, a tarefa `bundle` será executada em vez de `assemble` ([#1913](https://github.com/react-native-community/cli/pull/1913)).
- Remoção do fluxo de fallback para padrões de configuração do Metro ([#1972](https://github.com/react-native-community/cli/pull/1972)).
  - O [formato atualizado de `metro.config.js`](/blog/2023/06/21/0.72-metro-package-exports-symlinks#new-metroconfigjs-setup) da versão 0.72 agora é obrigatório na versão 0.73, pois removemos a cópia de fallback desses padrões do CLI.
- Remoção da opção `--configuration` do `run-ios` (substituída por `--mode`) ([#2028](https://github.com/react-native-community/cli/pull/2028)).
- Remoção da opção `--variant` do comando `build-android` (substituída por `--mode`) ([#2026](https://github.com/react-native-community/cli/pull/2026)).

[Veja o changelog completo para v12.0.0](https://github.com/react-native-community/cli/releases/tag/v12.0.0).

### Descontinuação do `@types/react-native`

Como mencionado em [First-class Support for TypeScript](/blog/2023/01/03/typescript-first#declarations-shipped-with-react-native), estamos enviando tipos TypeScript com `react-native` desde a versão 0.71 e agora estamos descontinuando `@types/react-native` para a versão 0.73.

Não lançaremos nenhum patch futuro para versões existentes. A orientação é migrar do `@types/react-native`. Veja instruções sobre [como migrar](/blog/2023/01/03/typescript-first#how-to-migrate).
