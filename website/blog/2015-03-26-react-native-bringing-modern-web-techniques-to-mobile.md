Nós introduzimos o [React](https://code.facebook.com/projects/176988925806765/react/) ao mundo há dois anos, e desde então ele teve um crescimento impressionante, tanto dentro quanto fora do Facebook. Hoje, mesmo que ninguém seja forçado a usá-lo, novos projetos web no Facebook são comumente construídos usando React de uma forma ou outra, e está sendo amplamente adotado em toda a indústria. Engenheiros estão escolhendo usar React todos os dias porque ele permite que eles passem mais tempo focando em seus produtos e menos tempo lutando com seu framework. Não foi até que estivéssemos construindo com React por um tempo, no entanto, que começamos a entender o que o torna tão poderoso.

React nos força a dividir nossas aplicações em componentes discretos, cada um representando uma única view. Esses componentes tornam mais fácil iterar em nossos produtos, já que não precisamos manter o sistema inteiro em nossas cabeças para fazer mudanças em uma parte dele. Mais importante, no entanto, React envolve a API mutável e imperativa do DOM com uma declarativa, o que eleva o nível de abstração e simplifica o modelo de programação. O que descobrimos é que quando construímos com React, nosso código é muito mais previsível. Esta previsibilidade faz com que possamos iterar mais rapidamente com confiança, e nossas aplicações são muito mais confiáveis como resultado. Além disso, não é apenas mais fácil escalar nossas aplicações quando elas são construídas com React, mas descobrimos que também é mais fácil escalar o tamanho de nossas próprias equipes.

Junto com o ciclo rápido de iteração da web, pudemos construir alguns produtos incríveis com React, incluindo muitos componentes do Facebook.com. Além disso, construímos frameworks incríveis em JavaScript em cima do React, como [Relay](https://reactjs.org/blog/2015/02/20/introducing-relay-and-graphql.html), que nos permite simplificar muito nossa busca de dados em escala. É claro que a web é apenas parte da história. O Facebook também tem aplicativos Android e iOS amplamente usados, que são construídos em cima de pilhas de tecnologia proprietárias e desconexas. Ter que construir nossos aplicativos em cima de múltiplas plataformas bifurcou nossa organização de engenharia, mas isso é apenas uma das coisas que torna o desenvolvimento de aplicativos móveis nativos difícil.
